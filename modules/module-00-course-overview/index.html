This course is built on the industry-leading open-source Observability stack. We focus on tools provided by the Cloud Native Computing Foundation (CNCF).

Prometheus üìà

The primary time-series database and metric collection engine. It uses a pull model to scrape metrics from targets at regular intervals.

Loki üíæ

The logging component, designed to be cost-effective and highly scalable. Unlike ElasticSearch, Loki indexes only metadata (labels), not the full log text, making querying and storage extremely efficient.

Grafana üñ•Ô∏è

The visualization layer. Grafana acts as the powerful front-end, connecting to both Prometheus and Loki to build integrated, actionable dashboards for all telemetry data.

OpenTelemetry (OTel) üîó

The instrumentation standard. OTel provides the APIs, SDKs, and collectors to generate and manage metrics, logs, and traces in a standardized, vendor-neutral way.

2. Understanding: Context and Course Flow üí°

The Modern Web Problem

In the past, web applications were monolithic and predictable. Today, they are distributed across microservices, containers (like Docker), and serverless functions. When a user experiences an error, pinpointing the source across dozens of interdependent services becomes nearly impossible using traditional monitoring alone.

History and Context

Observability as a concept originated in Control Theory and was popularized in the software world by teams at Google, Twitter, and Netflix. It is defined as the measure of how well internal states of a system can be inferred from its external outputs. For software, this means instrumenting the code to generate data (telemetry) that allows us to ask any question about the system *post-facto*.

Monitoring asks: "Is the system working?" (Predefined questions). Observability asks: "Why is the system not working?" (Unforeseen questions).

Benefits and Challenges

  • Benefits: Faster Mean Time To Resolution (MTTR), greater confidence in deploying code, and a deeper understanding of user experience.
  • Challenges: Requires initial code instrumentation, can generate massive amounts of data (cost management), and requires shifting the engineering mindset from reactive to proactive.

Course Flow

The course is structured to follow the life cycle of telemetry data:

  1. Foundations (M0-M1): Defining observability and telemetry concepts.
  2. Collection & Storage (M2-M3): Focus on instrumentation and getting data into the backend systems (Prometheus, Loki).
  3. Visualization & Action (M4-M5): Building dashboards in Grafana and setting up alerting.
  4. Project (M6): Applying all concepts to a real-world, end-to-end web application.

6. Creating: Future Potential and New Solutions üöÄ

The field of observability is constantly evolving. After mastering this stack, you will be prepared for several advanced directions:

Expanding Data Collection

  • Evolving OpenTelemetry: OTel is rapidly becoming the single standard for instrumentation. Future work involves extending OTel to more languages and advanced data correlation features.
  • Cloud Integration: Migrating the local Docker stack to managed services like AWS CloudWatch, Azure Monitor, or Google Cloud Operations (all of which integrate with the technologies learned here).

Advanced Analytics

Beyond simple alerting, the next step involves using Machine Learning to automatically detect anomalies and predict system failures based on the metric data you are now collecting. The principles of time-series analysis you learn here are the foundation for these advanced systems.
