1. What is telemetry in web applications?

In a web application, telemetry is the automatic collection of data about how the app behaves and how users interact with it. Instead of manually checking logs or databases, telemetry sends structured signals to your observability tools.

Examples of telemetry from a web app:

  • ‚ÄúUser clicked the Sign up button‚Äù (event)
  • ‚ÄúAPI response time is 320 ms‚Äù (metric)
  • ‚ÄúThis checkout request went through 4 services‚Äù (trace)

Telemetry is the raw material that observability tools like Prometheus, Grafana, and OpenTelemetry work with.

2. Client-side vs server-side telemetry

Web apps typically produce telemetry in two places:

Client-side telemetry

Collected in the browser. Measures how the page loads and how the user interacts with the UI.

  • Page load time in the user‚Äôs browser
  • Clicks, scroll depth, navigation
  • JavaScript errors

Server-side telemetry

Collected in your backend services. Measures how your APIs and services behave.

  • Request rate and error rate
  • Latency of API calls
  • Database query time

A page might load fast on the server but slow in the browser because of large images or blocking JavaScript. Client-side telemetry reveals that; server-side alone does not.

3. Events, metrics & traces in web apps

In Module 2 we saw the three pillars of observability. Here‚Äôs how they show up specifically in web apps:

Events

Discrete user actions like clicking buttons, submitting forms, or viewing specific pages. Example: ‚ÄúUser clicked ‚ÄòAdd to cart‚Äô with product_id=123.‚Äù

Metrics

Aggregated numbers over time. Example: ‚ÄúSign ups per minute‚Äù or ‚Äúp95 page load time‚Äù.

Traces

Show how a single request flows through your system, often starting in the browser and continuing through multiple services.

In practice, your app will often:

  • Track events for user behavior
  • Send metrics to Prometheus/Grafana
  • Generate traces via OpenTelemetry

4. Real User Monitoring (RUM)

Real User Monitoring (RUM) collects performance and behavior data from real users in real browsers.

RUM can tell you things like:

  • How long your app takes to load on users‚Äô devices
  • Which browsers or regions see more errors
  • How front-end performance changes after a deployment

RUM complements server-side telemetry. Together, they show both what the user experiences and how the system behaves internally.

5. Telemetry flow: browser ‚Üí backend ‚Üí collector

At a high level, telemetry from a web app often flows like this:

  1. The browser (client) sends events/metrics/traces to your backend or directly to a collector endpoint.
  2. The backend or collector normalizes and enriches the data.
  3. The observability stack stores and visualizes it (Prometheus, Grafana, trace UIs, etc.).

In later modules, we will build a simple version of this pipeline: browser ‚Üí backend ‚Üí Prometheus ‚Üí Grafana.

6. What to measure (and what not to)

Good telemetry is intentional. More data is not always better.

Things you usually do want to measure:

  • Page load and interaction performance
  • Key user actions aligned with your goals (sign up, checkout, etc.)
  • Errors and failed requests

Things to be careful about:

  • Avoid sending sensitive personal data in telemetry
  • Don‚Äôt log entire request bodies unnecessarily
  • Be mindful of volume and cost (too many events can get expensive)

Instead of logging full credit-card numbers, log a masked version or just an ID that can‚Äôt identify the card on its own.
